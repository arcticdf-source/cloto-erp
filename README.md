# CLOTO-NEW

Статический HTML/JS-проект для локального просмотра и публикации через GitHub.

## Продолжение работы в VS Code

Да, после публикации на GitHub вы продолжаете работать здесь же, в этом же проекте, через VS Code.
Обычный процесс такой:

1. Открываете папку проекта в VS Code.
2. Вносите изменения в HTML/JS-файлы.
3. Проверяете локально.
4. Делаете `git add`, `git commit`, `git push`.
5. GitHub обновляет удаленную версию проекта.

GitHub в этом случае нужен как удаленное хранилище и точка публикации, а не как замена вашей локальной работы.

## Локальный просмотр

Из папки проекта можно запустить один из вариантов:

### Вариант 1: через Node.js

```powershell
npm run serve
```

Сайт будет доступен по адресу:

```text
http://localhost:8080
```

### Вариант 2: через Python

```powershell
npm run serve:python
```

или напрямую:

```powershell
python -m http.server 8080
```

## Публикация на GitHub

1. Создайте пустой репозиторий на GitHub, например `CLOTO-NEW`.
2. Привяжите локальный проект к удаленному репозиторию:

```powershell
git remote add origin https://github.com/<username>/CLOTO-NEW.git
git branch -M main
git push -u origin main
```

3. На GitHub откройте `Settings -> Pages`.
4. В `Build and deployment` выберите:
   - `Source: Deploy from a branch`
   - `Branch: main`
   - `Folder: / (root)`
5. Сохраните настройки и дождитесь публикации.

После этого сайт обычно будет доступен по адресу вида:

```text
https://<username>.github.io/CLOTO-NEW/
```

## Защищенный просмотр по логину и паролю

Публичный GitHub-репозиторий не дает реальной защиты содержимого паролем. Если репозиторий публичный, файлы репозитория доступны всем.

Если нужен именно сайт по ссылке с запросом логина и пароля, используйте защищенный сервер предпросмотра в этом проекте.

Этот же вариант теперь подходит и для загрузки рабочих файлов по клиентам: карточка клиента умеет отправлять документы на сервер, а сервер сохраняет их на диск в папку `uploads/client-files`.

### Запуск в PowerShell

```powershell
$env:AUTH_USER='demo'
$env:AUTH_PASS='strong-password'
npm run serve:auth
```

После этого сайт будет доступен по адресу:

```text
http://localhost:8080
```

При открытии браузер запросит логин и пароль.

### Что это решает

- Сайт открывается только после ввода логина и пароля.
- Можно поднять его на своем компьютере, сервере или VPS.
- Можно загружать файлы из карточек клиентов на сервер, а не в `localStorage` браузера.
- Это подходит для внутреннего показа команде.

### Что это не решает

- Это не делает публичный GitHub-репозиторий приватным.
- Если вы оставите репозиторий публичным на GitHub, исходные файлы и документы из репозитория все равно будут видны.
- Один только HTTP с Basic Auth не считается безопасным размещением для интернета. Для внешнего доступа нужен HTTPS через reverse proxy или платформу с TLS.

## Рекомендуемая схема безопасного размещения

Для вашей текущей задачи лучший вариант не GitHub Pages и не открытый `443 -> Node`, а внешний VPS с zero-trust-доступом для сотрудников.

Рекомендуемый production-вариант:

1. VPS на `Ubuntu 24.04+`.
2. Приложение в Docker.
3. Приложение слушает только внутреннюю сеть контейнеров.
4. Внешний доступ идет через `Cloudflare Tunnel`.
5. Доступ сотрудников ограничивается через `Cloudflare Access` по email или корпоративному домену.
6. Внутри приложения остается второй рубеж: `Basic Auth`.

Это дает удаленный браузерный доступ для сотрудников без локального сервиса и без прямой публикации порта приложения в интернет.

Готовые файлы для такого варианта лежат здесь:

1. `deploy/docker-compose.zero-trust.example.yml`
2. `deploy/.env.external.example`
3. `deploy/SECURE_EXTERNAL_DEPLOY.md`

Для рабочего варианта с загрузкой файлов лучше не GitHub Pages, а обычный VPS или внутренний сервер.

Рекомендуемый стек:

1. VPS на `Ubuntu 22.04+` или внутренний Windows/Linux сервер.
2. Node.js `20 LTS`.
3. Этот проект с запуском `npm run serve:auth`.
4. Reverse proxy перед Node.js:
   - `Caddy` как самый простой вариант с автоматическим HTTPS.
   - или `Nginx` + Let's Encrypt.
5. Ограничение доступа:
   - Basic Auth уже есть в приложении.
   - при необходимости сверху можно добавить VPN, IP allowlist или SSO на уровне reverse proxy.

Почему это лучше:

- файлы попадают на диск сервера, а не в браузер;
- нет лимитов `localStorage`;
- можно делать резервные копии папки `uploads/`;
- сайт остается таким же HTML/JS, без тяжелой переделки фронтенда.

## Быстрый деплой на VPS

Примерная последовательность:

1. Установить Node.js `20 LTS`.
2. Скопировать проект на сервер.
3. Выполнить:

```bash
npm install
export AUTH_USER='demo'
export AUTH_PASS='strong-password'
export PORT='8080'
export HOST='127.0.0.1'
npm run serve:auth
```

4. Поднять reverse proxy на `443` с HTTPS и проксированием на `http://127.0.0.1:8080`.
5. Настроить автозапуск через `pm2`, `systemd` или NSSM/Task Scheduler на Windows.

Если нужен именно внешний безопасный доступ сотрудников, используйте не этот быстрый вариант, а zero-trust-схему из `deploy/SECURE_EXTERNAL_DEPLOY.md`.

## Где лежат файлы

После загрузки из карточки клиента файлы сохраняются в:

```text
uploads/client-files/<Компания>/<Папка>/
```

Для текущего кейса папка называется:

```text
Требования к химреагентам и материалам
```

## Что лучше выбрать сейчас

Если цель — быстро получить рабочую защищенную ссылку для команды, практичный вариант такой:

1. Поднять этот проект на VPS.
2. Не открывать порт приложения наружу.
3. Давать доступ сотрудникам через `Cloudflare Tunnel + Cloudflare Access`.
4. Оставить `Basic Auth` внутри приложения как второй рубеж.
5. Делать бэкап папки `uploads/`.

Это даст и защищенный доступ, и нормальную загрузку файлов без переписывания продукта заново.

## Если нет VPS и домена

Если у вас сейчас нет ни VPS, ни домена, то лучший безопасный вариант другой:

1. выделенный всегда включенный ПК или мини-ПК,
2. запуск CLOTO на нем,
3. доступ сотрудников только через `Tailscale`,
4. `Basic Auth` внутри приложения,
5. резервные копии папки `uploads/`.

Подробная инструкция лежит в `deploy/NO_VPS_NO_DOMAIN.md`.

Почему именно так:

1. не нужен домен,
2. не нужен VPS,
3. сайт не публикуется в открытый интернет,
4. доступ можно выдавать только сотрудникам.

Если же вам принципиально нужен публичный URL без установки Tailscale-клиента сотрудникам, тогда без VPS/домена останется только managed-хостинг вроде Render или Railway, но это будет менее безопасно, чем Tailscale-схема.

## Временный внешний тестовый сервер

Если задача сейчас такая:

1. временно вынести сайт наружу,
2. тестировать его на внешнем URL,
3. продолжать наполнять и редактировать проект локально в VS Code,
4. а потом перенести все на локальный выделенный ПК,

то это нормальный рабочий путь.

Для такого сценария в проект добавлен вариант под Render:

1. `render.yaml`
2. `package.json` script `start`
3. `deploy/RENDER_TEST_DEPLOY.md`

Как это работает:

1. Вы редактируете проект локально в VS Code.
2. Делаете `git commit` и `git push`.
3. Render автоматически обновляет внешний тестовый сервер.
4. Позже тот же проект переносится на локальный выделенный ПК.

Важно: такой внешний managed-хостинг подходит именно для временного тестового контура. Дисковое хранение файлов на нем нельзя считать постоянным без отдельного persistent storage.

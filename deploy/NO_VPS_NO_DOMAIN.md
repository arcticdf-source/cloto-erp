# Размещение CLOTO без VPS и без домена

## Короткий вывод

Если у вас нет VPS и нет домена, то самый безопасный вариант для удаленного доступа сотрудников такой:

1. Выделить один постоянно включенный компьютер или мини-ПК.
2. Запускать CLOTO на нем локально.
3. Давать доступ сотрудникам только через `Tailscale`.
4. Не открывать сайт в публичный интернет вообще.

Это не требует домена, не требует VPS и при этом намного безопаснее, чем любой публичный бесплатный хостинг с открытой ссылкой.

## Почему это лучший вариант при ваших ограничениях

1. Сайт не будет доступен всему интернету.
2. Каждый сотрудник получает доступ только после входа в tailnet.
3. Можно быстро отключить любого сотрудника.
4. Файлы продолжают храниться у вас, а не у стороннего хостинга.
5. Не нужен домен и не нужен reverse proxy.

## Что нужно

1. Один Windows-ПК или мини-ПК, который будет всегда включен.
2. Стабильный интернет.
3. Tailscale-аккаунт.
4. Список сотрудников, которым нужен доступ.

## Схема работы

1. На выделенном ПК запускается `npm run serve:auth`.
2. На этот ПК ставится Tailscale.
3. Сотрудники тоже ставят Tailscale.
4. Доступ к сайту идет по tailnet-адресу вида `http://100.x.x.x:8080` или по MagicDNS-имени.
5. Внутри сайта остается Basic Auth как второй рубеж.

## Пошаговая настройка

### 1. Подготовить выделенный хост

Лучше использовать:

1. отдельный офисный ПК,
2. мини-ПК,
3. или Windows Server внутри компании.

Не рекомендуется использовать личный ноутбук, который часто выключается.

### 2. Установить Node.js

Поставьте Node.js `20 LTS`.

### 3. Подготовить проект

Из папки проекта:

```powershell
npm install
```

### 4. Запускать приложение только на интерфейсе Tailscale или на всех интерфейсах внутри сети

Пример для Windows PowerShell:

```powershell
$env:AUTH_USER='demo'
$env:AUTH_PASS='strong-password'
$env:PORT='8080'
$env:HOST='0.0.0.0'
$env:TRUST_PROXY='0'
npm run serve:auth
```

### 5. Установить Tailscale на хост

После установки:

1. войдите под админским аккаунтом,
2. включите устройство в tailnet,
3. проверьте выданный IP или MagicDNS-имя.

### 6. Ограничить доступ сотрудникам

В Tailscale admin console:

1. добавьте только нужных сотрудников,
2. включите MFA у аккаунтов,
3. удаляйте доступ у тех, кому он больше не нужен.

### 7. Открывать сайт сотрудникам только по tailnet-адресу

Примеры:

```text
http://100.x.x.x:8080
```

или

```text
http://cloto-host.tailnet-name.ts.net:8080
```

## Автозапуск на Windows

Чтобы сайт поднимался сам после перезагрузки, лучше использовать `nssm` или Windows Task Scheduler.

Минимально надежный вариант: Task Scheduler, который запускает PowerShell-скрипт с env-переменными и `npm run serve:auth`.

## Файлы и резервные копии

Файлы клиентов лежат в:

```text
uploads/client-files
```

Минимум раз в день эту папку нужно копировать:

1. на второй диск,
2. в сетевую папку,
3. или в облако компании.

## Что хуже, но возможно

Если нужен именно внешний публичный URL без VPS и без домена, возможны managed-платформы:

1. Render
2. Railway
3. Fly.io

Они дадут адрес вида `*.onrender.com` или `*.up.railway.app`, но это уже публичный интернет-адрес. Без отдельного домена и внешней системы доступа вы будете опираться в основном на Basic Auth приложения. Это заметно слабее, чем Tailscale.

## Итоговая рекомендация

Если приоритет — безопасность, а не публичная ссылка, выбирайте:

1. выделенный хост,
2. Tailscale,
3. Basic Auth внутри приложения,
4. ежедневные бэкапы `uploads`.

Если приоритет — быстро дать всем ссылку без установки клиента, тогда уже нужен либо VPS+домен, либо managed-хостинг с публичным URL.

# Временный внешний тестовый сервер без VPS

## Подходит ли это вам

Да. Для тестирования можно временно разместить CLOTO на внешнем managed-хостинге, а позже перенести тот же проект на локальный выделенный ПК.

Самое важное:

1. Вы продолжаете редактировать проект локально в VS Code.
2. Внешний сервер просто забирает обновления из GitHub.
3. Когда все будет готово, переносите этот же код на локальный хост.

## Как вы будете дальше работать в VS Code

Рабочий цикл будет такой:

1. Открываете проект локально в VS Code.
2. Меняете HTML, JS, CSS и серверный код.
3. Проверяете локально.
4. Делаете `git add`, `git commit`, `git push`.
5. Render автоматически обновляет внешний тестовый сервер.

То есть Visual Studio Code остается вашим основным местом работы. Ничего не переносится в браузерный редактор и не требует ручного редактирования на хостинге.

## Почему Render подходит для тестового этапа

1. Не нужен VPS.
2. Не нужен свой домен.
3. Есть внешний URL вида `https://...onrender.com`.
4. Есть автодеплой из GitHub.
5. Есть TLS/HTTPS из коробки.

## Ограничение, которое нужно понимать сразу

Для тестового этапа Render подходит, но хранение файлов на диске сервера надо считать временным.

Причина:

1. локальный filesystem на managed-хостинге может быть непостоянным,
2. после redeploy или миграции инстанса загруженные файлы могут потеряться,
3. значит этот вариант годится для интерфейса, логики и сценариев работы, но не как финальное постоянное хранилище документов.

Если нужно тестировать загрузку файлов без риска потери, позже надо будет добавить внешнее object storage. Пока для первого теста можно использовать этот вариант как временный.

## Что уже подготовлено в проекте

В проект добавлены:

1. `render.yaml`
2. `package.json` со script `start`

Это достаточно, чтобы Render поднял приложение как Node web service.

## Пошаговый запуск на Render

### 1. Загрузить проект в GitHub

Если репозиторий еще не загружен:

```powershell
git init
git add .
git commit -m "Initial deploy setup"
git branch -M main
git remote add origin https://github.com/<username>/<repo>.git
git push -u origin main
```

### 2. Создать Web Service в Render

В Render:

1. New +
2. Web Service
3. Connect repository
4. Выбрать ваш GitHub-репозиторий

### 3. Указать настройки

Если Render подхватит `render.yaml`, сервис создастся почти автоматически.

Если вводить вручную:

1. Runtime: Node
2. Build Command: `npm install`
3. Start Command: `npm start`

### 4. Заполнить переменные окружения

Обязательно задать:

1. `AUTH_USER`
2. `AUTH_PASS`

Остальное уже предусмотрено в `render.yaml`.

### 5. Дождаться деплоя

После деплоя Render даст URL вида:

```text
https://cloto-new-test.onrender.com
```

## Как обновлять сайт дальше

Каждое следующее обновление делается так же из VS Code:

```powershell
git add .
git commit -m "Update portal"
git push
```

После `git push` Render автоматически запустит redeploy.

## Что лучше не делать на этом этапе

1. Не считать Render постоянным файловым хранилищем.
2. Не загружать туда критичные документы как единственную копию.
3. Не хранить секреты в репозитории.

## Что потом будет при переносе на локальный выделенный ПК

Перенос будет прямолинейный:

1. берете тот же Git-репозиторий,
2. клонируете его на выделенный ПК,
3. запускаете `npm install` и `npm run serve:auth`,
4. выдаете доступ сотрудникам через Tailscale,
5. внешний тестовый сервер выключаете.

То есть временный внешний сервер не мешает дальнейшему переходу на локальный выделенный хост.

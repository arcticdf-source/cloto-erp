# Безопасное внешнее размещение CLOTO

## Рекомендуемый вариант

Для вашей ситуации лучший вариант такой:

1. Отдельный VPS на Ubuntu 24.04.
2. Приложение запускается в Docker-контейнере.
3. Снаружи сайт не публикует порт приложения вообще.
4. Доступ сотрудников идет через Cloudflare Tunnel.
5. Доступ в сам сайт ограничивается Cloudflare Access по списку сотрудников.
6. Внутри приложения остается второй рубеж защиты: Basic Auth.
7. SSH к серверу открыт только для администратора по ключу и только с доверенного IP или через Tailscale.

Это безопаснее обычного варианта с публичным `443 -> reverse proxy -> node`, потому что приложение не висит открытым в интернете и не принимает прямой трафик извне.

## Почему этот вариант лучше

1. У приложения нет публичного порта.
2. Сотрудникам не нужен VPN-клиент: достаточно браузера.
3. Можно пускать только конкретные email-адреса или домен компании.
4. Можно включить MFA на уровне Cloudflare Access.
5. Файлы клиентов продолжают храниться на сервере в `uploads/client-files`.
6. Бэкапы делаются обычным копированием папки `uploads`.

## Схема доступа

1. Сотрудник открывает защищенный домен, например `https://portal.example.com`.
2. Cloudflare Access проверяет email, SSO или одноразовый код.
3. Только после этого трафик попадает в Cloudflare Tunnel.
4. Tunnel передает запрос во внутренний контейнер `app`.
5. Приложение дополнительно требует свой логин и пароль.

## Что нужно подготовить

1. VPS с Ubuntu 24.04 и статическим IP.
2. Домен, который можно отдать под портал.
3. Аккаунт Cloudflare Zero Trust.
4. Список сотрудников, кому нужен доступ.
5. Отдельный длинный пароль для внутреннего Basic Auth.

## Что нельзя делать

1. Не публиковать `8080` наружу в firewall или в Docker.
2. Не оставлять SSH по паролю.
3. Не использовать один общий слабый пароль для всех сотрудников.
4. Не хранить `.env.external` в репозитории.
5. Не хранить бэкапы на том же сервере без копии во внешнее хранилище.

## Пошаговый деплой

### 1. Подготовить сервер

```bash
sudo apt update
sudo apt install -y ca-certificates curl ufw
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
```

После этого перелогиньтесь на сервер.

### 2. Закрыть лишние порты

Пример минимального firewall:

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow from YOUR_ADMIN_IP to any port 22 proto tcp
sudo ufw enable
sudo ufw status verbose
```

Порт `8080` открывать не нужно.

### 3. Разместить проект

```bash
cd /opt
sudo mkdir -p CLOTO-NEW
sudo chown $USER:$USER CLOTO-NEW
```

Дальше загрузите содержимое проекта в `/opt/CLOTO-NEW`.

### 4. Подготовить env

```bash
cd /opt/CLOTO-NEW/deploy
cp .env.external.example .env.external
```

Заполните минимум такие значения:

1. `AUTH_USER`
2. `AUTH_PASS`
3. `CLOUDFLARE_TUNNEL_TOKEN`
4. `HOST=0.0.0.0`
5. `TRUST_PROXY=1`

## 5. Создать Cloudflare Tunnel

В Cloudflare Zero Trust:

1. Создайте Tunnel.
2. Привяжите hostname, например `portal.example.com`.
3. В качестве target укажите сервис `http://app:8080`.
4. Скопируйте tunnel token и вставьте его в `.env.external`.

## 6. Ограничить доступ сотрудникам

В Cloudflare Access создайте приложение типа Self-hosted и правило доступа:

1. Разрешить только конкретные email сотрудников.
2. Или разрешить только корпоративный email-domain.
3. Включить MFA.
4. Запретить доступ всем остальным по default deny.

## 7. Запустить стек

```bash
cd /opt/CLOTO-NEW/deploy
docker compose -f docker-compose.zero-trust.example.yml --env-file .env.external up -d --build
```

## 8. Проверить

Проверки должны быть такие:

1. Без Cloudflare Access сайт не открывается.
2. После Access открывается логин приложения.
3. После входа доступна карточка клиента.
4. Загрузка файлов работает.
5. На сервере файлы появляются в `uploads/client-files/...`.

## Резервное копирование

Минимум один ежедневный backup папки `uploads` во внешнее хранилище:

```bash
tar -czf /tmp/cloto-uploads-$(date +%F).tar.gz /opt/CLOTO-NEW/uploads
```

Дальше архив нужно отправлять в S3-совместимое хранилище, Backblaze B2, Яндекс Object Storage или другой внешний backup target.

## Администрирование

Рекомендуемый порядок:

1. Доступ сотрудников только через Cloudflare Access.
2. Доступ администратора на сервер только по SSH-ключу.
3. Лучше дополнительно завернуть админский доступ в Tailscale.
4. Обновления делать через `docker compose up -d --build`.

## Если нужен еще более строгий режим

Самый строгий вариант еще безопаснее, но менее удобен для сотрудников:

1. Не Cloudflare Access, а Tailscale.
2. Портал доступен только участникам tailnet.
3. Каждый сотрудник ставит Tailscale-клиент.

Этот вариант самый закрытый, но для массового браузерного доступа сотрудников обычно менее удобен, чем Cloudflare Access.
